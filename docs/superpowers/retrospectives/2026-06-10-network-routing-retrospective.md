# 2026-06-10 网络 / 路由 / OpenWrt / Passwall 复盘

## 背景

本轮任务连续覆盖了上游同步、发布、服务器部署、面板验证、线上入站审计、OpenWrt/Passwall 分流重构、AI 域名清单治理等多个工作域，期间暴露出若干运行态排障的典型陷阱。

## 关键误判

### 1. 把节点本身当成首要嫌疑

最初容易把体验问题直接归因到 `85.155.178.115`，但后续证据显示：

- 服务器自身出口正常
- OpenWrt 与客户端路径存在透明代理 / 默认链 / ACL 串扰
- 真正需要拆的是路径，而不是先换节点

### 2. 把“切换成功”误判为“出口已切换”

出现过：

- `uci` 中 `tcp_node` 已切换
- Passwall 运行进程也换成了新配置
- 但客户端 `ipify` 仍是旧出口

最终证明必须同时看：

- `ipify`
- `/tmp/etc/passwall/var`
- `iptables -vnL`
- Passwall/Xray 运行参数

### 3. 混淆显式 SOCKS 与透明代理

OpenWrt 本地 SOCKS 1081 可以正常出口 85，不代表主 WiFi 透明代理路径一定已经切到 85。两者需要分开验证。

## 本轮正确做法

- 先把 `USA-US` 与主 WiFi 分工拆清
- 再给 `lan` 增加独立 ACL
- 把 AI / GFW 域名最小集合加入 `proxy_host`
- 保持 `USA-US` 继续走 `35.87.239.230`
- 通过 `iptables` 命中和 Passwall 运行缓存确认主 WiFi 规则已生效

## 沉淀出的最佳实践

1. 主 WiFi 默认直连，只把高价值 AI / GFW 域名送入稳定出口。
2. 单独的 USA WiFi 继续保留为全代理网络，避免污染主 WiFi 的日常体验。
3. 模型接口域名要单独维护，不能只看站点首页域名。
4. 网络 / 路由问题必须用双视角证据闭环。
