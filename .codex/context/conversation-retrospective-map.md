# 对话复盘地图：运行态网络、部署与治理陷阱

## 目的

沉淀近期多轮发布、部署、服务器审计、OpenWrt/Passwall 分流、AI 域名治理任务中暴露的误判模式、认知盲区和执行偏差，供后续任务先读后动。

## 高频陷阱

### 1. 把“静态配置已修改”误当成“运行路径已生效”

- 典型表现：`uci` / 配置文件已切到新节点，但客户端 `ipify` 仍是旧出口。
- 修正动作：优先读取 `iptables -vnL` 计数、`/tmp/etc/passwall/var`、Passwall/Xray 运行进程参数，再决定是否真的生效。

### 2. 把“页面可访问”误当成“模型接口可用”

- 典型表现：站点首页、控制台能打开，但 API、IDE 插件、流式响应失败。
- 修正动作：分开验证主站、控制台、API、静态资源、插件代理域名。

### 3. 混淆显式 SOCKS、透明 REDIRECT、TPROXY 和 DNS 劫持

- 典型表现：用单个 `curl -x socks5h://...` 的结果，推断整条透明代理链。
- 修正动作：明确当前验证的是哪一层路径；显式 SOCKS 成功不代表透明 ACL 一定成功。

### 4. 在任务域切换时不重述“当前主问题”

- 典型表现：把上游同步、构建发布、线上部署、路由器审计、AI 域名治理混成一个大问题。
- 修正动作：跨域前先写出“当前主问题 / 暂挂项 / 证据边界 / 回滚边界”。

### 5. 用单视角证据下结论

- 典型表现：只看客户端、只看路由器、或只看服务器某一侧。
- 修正动作：网络和分流问题至少要求双视角证据：客户端 + 中间设备，或中间设备 + 服务器。

## 运行态证据优先级

1. 客户端观测到的实际出口/失败模式。
2. 中间设备（OpenWrt/Passwall）ACL 计数、运行缓存、进程参数。
3. 服务器侧真实运行进程、端口、出口与日志。
4. 静态配置文件和设计预期。

## 结论模板

对网络与代理类任务，最终结论至少回答：

- 当前客户端在什么网段/SSID/接口。
- 实际命中的是哪条 ACL / routing rule。
- 真实出口是谁。
- 当前问题属于客户端、中间设备、服务器还是配置设计。
- 回滚点是什么。
